# AI Workflow Rules — Blood Donation App

These are direct rules for the AI coding agent. They are not suggestions. Follow them on every unit, every session, without exception.

---

## Before You Touch Any Code

1. Read ALL six context files in the order listed in `CLAUDE.md` before writing a single line
2. Read the spec file for the current unit completely before starting implementation
3. If the spec contradicts a context file, stop and report the conflict — do not resolve it yourself
4. If a requirement in the spec is ambiguous, ask for clarification before implementing

---

## Scoping Rules

1. Work on exactly one unit at a time — the unit listed in the current spec file
2. Do not implement features, routes, components, or database changes that belong to a future unit
3. Do not refactor code from a previous unit unless the current spec explicitly requires it
4. Do not install any package not listed in the current spec's Dependencies section
5. Do not modify `supabase/migrations/` files that have already been applied — create new migration files instead
6. Do not modify files in the UI component library that were generated by a tool (e.g., NativeWind preset files)

---

## Implementation Rules

1. Follow `context/code-standards.md` exactly — naming, patterns, imports, and styling rules apply to every file
2. Use only the stack defined in `context/architecture.md` — do not introduce alternative libraries
3. All Supabase calls go through Zustand store actions — never call Supabase directly from a screen component
4. All blood group compatibility checks must use `lib/blood-compat.ts` — no inline logic
5. Eligibility is read from `donors.is_eligible` (generated column) — never compute it manually on the client
6. All forms must use React Hook Form + Zod — no uncontrolled inputs
7. All screens must handle: loading state, empty state, and error state
8. All tap targets must be a minimum of 44px in height

---

## When Requirements Are Missing

1. If a spec section is underspecified, implement the simplest correct version and leave a `// TODO:` comment marking the decision
2. If a database column is needed that is not in the schema, stop and report — do not alter the schema without explicit instruction
3. If a navigation route is needed that is not in the architecture, stop and report — do not add routes not in the file structure
4. If a new package is needed that is not in the spec, stop and report the specific package and why it's needed

---

## Files You Must Not Touch Without Explicit Instruction

- `context/*.md` — update `progress-tracker.md` only, after completing a unit
- `supabase/migrations/*.sql` — append new files only, never edit existing ones
- `lib/blood-compat.ts` — the compatibility matrix is fixed; do not modify the logic
- `types/index.ts` — you may add new types, but never remove or rename existing ones
- `lib/supabase.ts` — the client setup is set once; do not change it

---

## Verification Before Marking a Unit Complete

Run through this checklist before marking any unit done in `progress-tracker.md`:

- [ ] All acceptance criteria in the spec's "Verify when done" section are met
- [ ] TypeScript compiles with zero errors (`npx tsc --noEmit`)
- [ ] No `console.log` statements left in committed code
- [ ] No `any` types introduced
- [ ] All screens handle loading, empty, and error states
- [ ] NativeWind classes used — no inline styles
- [ ] All Supabase calls are in store actions
- [ ] `npx expo start` runs without errors

---

## Progress Tracker Updates

After completing a unit:
1. Move the unit from "In Progress" to "Completed" in `context/progress-tracker.md`
2. Move the next unit from "Next Up" to "In Progress"
3. If any architectural decisions were made during the unit, document them under "Architecture Decisions"
4. If any open questions were resolved, remove them from "Open Questions"
5. Add a brief session note under "Session Notes" describing what was built

---

## Handling Bugs and Corrections

When a bug is reported or a previous unit's output needs correction:
1. Read the original spec for that unit before touching the code
2. Fix only what the bug report specifies — do not refactor beyond the fix
3. After the fix, re-run the verification checklist for that unit
4. Document the fix in "Session Notes" of `progress-tracker.md`

---

## Splitting Units

If a unit turns out to be too large to complete in one session:
1. Complete everything up to a clean stopping point (a working sub-feature, not mid-implementation)
2. Document the stopping point clearly in "Session Notes"
3. List remaining work as a sub-item under "In Progress"
4. Resume from the documented stopping point in the next session

---

## What Done Looks Like

A unit is complete only when:
- Every item in the spec's "Verify when done" checklist passes
- The feature works end-to-end as described in the spec's Goal section
- No TypeScript errors exist
- The app compiles and runs without crashes related to this unit
- `progress-tracker.md` has been updated
